# Plan 01-02: Auth API Implementation

Design and build the secure, role-based login system for Teachers and Parents using JWT/OTP.

<objective>
To implement a production-ready authentication system that allows Teachers and Parents to sign in securely across multiple school tenants.
</objective>

<tasks>
<task>
<read_first> ["backend/src/models/BaseModel.js"] </read_first>
<action> Implement `backend/src/models/School.js` and `backend/src/models/User.js`. Define roles enum: `teacher`, `parent`, `student`, `admin`. </action>
<acceptance_criteria> `User.js` model handles password hashing (bcrypt) and multi-tenant `school_id` association. </acceptance_criteria>
</task>

<task>
<read_first> [] </read_first>
<action> Implement `backend/src/services/otpService.js` to simulate OTP (logging to console in v1 dev) with a clean service hook for MSG91 integration. </action>
<acceptance_criteria> `otpService` can generate a 6-digit code, store it temporarily (Redis/Mem/DB), and verify it. </acceptance_criteria>
</task>

<task>
<read_first> ["backend/src/models/User.js", "backend/src/services/otpService.js"] </read_first>
<action> Build `backend/src/controllers/authController.js` and `backend/src/routes/auth.js` for `POST /api/auth/login` (request OTP) and `POST /api/auth/verify-otp-password` (final login). </action>
<acceptance_criteria> `POST /api/auth/verify-otp-password` returns `{ access_token, refresh_token, user: { id, role, school_id } }` on success. </acceptance_criteria>
</task>

<task>
<read_first> ["backend/src/controllers/authController.js"] </read_first>
<action> Implement `POST /api/auth/refresh` to rotate the Access Token using a valid Refresh Token. </action>
<acceptance_criteria> Refresh endpoint works correctly and invalidates old tokens if required by security policy. </acceptance_criteria>
</task>

<task>
<read_first> ["backend/src/routes/auth.js"] </read_first>
<action> Add the FCM token registration endpoint `POST /api/auth/fcm-token` that associates the current user's device with their ID. </action>
<acceptance_criteria> FCM registration persists in the `user_devices` table. </acceptance_criteria>
</task>
</tasks>

<verification>
1. Use Postman/cURL to request an OTP for a predefined User.
2. Complete the login with OTP + Password and receive the JWT.
3. Test that the JWT payload includes the `school_id` and `role`.
</verification>

<must_haves>
- Secure token handling: No sensitive data in JWT payload.
- Password hashing: All passwords stored only as bcrypt hashes.
- multi-persona login: Support Teachers and Parents on the same endpoint via a `role` discriminator if needed, or separate routes.
</must_haves>
