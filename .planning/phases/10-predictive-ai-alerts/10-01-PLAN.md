# Plan 10-01: Risk Compute Engine

Implement the deterministic AI proxy that scans for Attendance, Academic, and Fee anomalies.

<objective>
To build a scalable background cron job that reads all active students against 3 threshold rules and stores flags in a new `risk_assessments` table for morning dispatches.
</objective>

<tasks>
<task>
<read_first> ["backend/src/models/BaseModel.js"] </read_first>
<action> Create `backend/src/models/RiskAssessment.js`. This model manages the `risk_assessments` table tracking `student_id`, `risk_type` (ATTENDANCE, ACADEMIC, FEE), `severity` (LOW, HIGH), and `description`. </action>
<acceptance_criteria> Table supports quick querying by teachers. </acceptance_criteria>
</task>

<task>
<read_first> ["backend/src/cron/meetingReminders.js"] </read_first>
<action> Create `backend/src/cron/riskEngine.js`. A function defined to run at 2:00 AM. It must:
1. Aggregate Attendance: Flag students < 75% or 3 consecutive absences.
2. Aggregate Results: Flag students averaging < 40% across published exams.
Store these flags or overwrite existing active flags in `RiskAssessment`. </action>
<acceptance_criteria> The logic accurately transforms large datasets into discrete risk insights. </acceptance_criteria>
</task>

<task>
<read_first> [] </read_first>
<action> Create `backend/src/cron/morningDispatcher.js`. Runs at 8:00 AM. It fetches all unnotified HIGH severity risks and calls `NotificationService.sendPush` to the parents, then marks them as notified. </action>
<acceptance_criteria> Prevents notification spam (e.g., only sends an alert once per risk trigger window). </acceptance_criteria>
</task>

<task>
<read_first> ["backend/src/app.js"] </read_first>
<action> Initialize both `riskEngine.js` and `morningDispatcher.js` in the main express file. </action>
<acceptance_criteria> Both workers load correctly on boot. </acceptance_criteria>
</task>
</tasks>

<verification>
1. confirm the `riskEngine` correctly flags a mock student with 3 absences.
2. verify `morningDispatcher` skips notifications if the risk was already acknowledged or notified within 7 days.
</verification>

<must_haves>
- Decoupled Dispatch: Splitting the heavy compute (2 AM) from the user-facing push (8 AM) is a non-negotiable architectural split.
- Fault Tolerance: If `riskEngine` fails on one school, it must continue looping through others.
</must_haves>
